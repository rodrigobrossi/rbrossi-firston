02 / 06 — BACKEND FOR FRONTEND (BFF)

Por que um BFF?

O frontend nunca fala diretamente com os microserviços. O BFF é a única porta de entrada — ele valida o JWT, injeta o user-id, aplica rate limiting, agrega dados e protege os serviços internos.

1
Autenticação centralizada
O BFF valida o JWT em todas as rotas /api/*. Os microserviços confiam no header x-user-id injetado pelo BFF — não precisam validar token individualmente.
2
Agregação de dados — GET /api/dashboard
Uma única chamada do frontend recebe dados de 3 serviços em paralelo (contatos, pipeline, calendário) via Promise.allSettled. Elimina waterfalls de rede no browser.
3
Rate limiting por camada
/auth/*: 30 req/15min · /api/*: 200 req/min. Protege contra abusos sem tocar nos serviços internos.
4
Circuit Breaker em cada downstream call
Cada serviço tem seu próprio circuit breaker via opossum. Se o billing cair, o BFF retorna fallback — o resto do app continua funcionando.
5
Visibilidade
GET http://localhost:4000/health retorna o estado (OPEN/CLOSED/HALF_OPEN) de todos os circuit breakers em tempo real.
03 / 06 — CIRCUIT BREAKER PATTERN

Circuit Breaker — como funciona

Implementado com a biblioteca opossum (Node.js). Cada serviço downstream tem um circuit breaker independente com janela deslizante de 10s.

CLOSED — normal
Chamadas passam normalmente.
O breaker monitora a taxa de falhas.

Abre quando: >50% de falhas
nas últimas 10 chamadas (mín. 3).
OPEN — curtocircuito
Chamadas são bloqueadas imediatamente.
O BFF retorna o fallback configurado
(ex: lista vazia, 503 com mensagem).

Fica aberto por: 15 segundos.
HALF-OPEN — sondagem
Após 15s, uma chamada de teste é liberada.
Se sucesso → volta a CLOSED.
Se falha → volta a OPEN por mais 15s.
// CONFIGURAÇÃO POR SERVIÇO (crm/services/bff/src/circuitBreaker.js)
timeout:                  5000   // 5s para considerar falha
errorThresholdPercentage:   50   // abre se >50% falhar
resetTimeout:             15000   // tenta recuperar após 15s
volumeThreshold:              3   // mínimo de chamadas antes de agir
rollingCountTimeout:      10000   // janela deslizante de 10s

05 / 06 — ESCALABILIDADE

Pronto para escalar

O design local já espelha a arquitetura de produção na AWS. Para escalar de 10 para centenas de usuários, são apenas 4 mudanças de variável de ambiente.

4
mudanças de env para ir a produção
~$116
USD/mês na AWS para até 500 usuários
18
usuários pagantes para break-even
88%
margem bruta com 100 usuários
↑
Escalar BFF horizontalmente
O BFF é stateless. Em produção, múltiplas tasks do ECS Fargate rodam atrás de um Application Load Balancer. Circuit breakers são por processo (em-memória), o que é correto para isolamento.
↑
Cada microserviço escala independente
Se sentiment vira gargalo, sobe só o sentiment. ECS Fargate com Auto Scaling por CPU/memória. Sem impacto nos outros serviços.
↑
MySQL → RDS Multi-AZ / Read Replicas
Leituras pesadas (relatórios, dashboard) podem ser redirecionadas para read replicas. Write vai sempre ao primary. Zero mudança de código — só variável de ambiente.
↑
Redis → ElastiCache cluster mode
Sessões, rate limit counters e cache de tokens já usam Redis. Em produção: ElastiCache Multi-AZ com failover automático em <60s.
